## 1. Why this exists

Today's iOS Listen tab works, but is hobbled in two ways:

1. **Single-locale lock-in.** `SFSpeechRecognizer` is hard-locked to one BCP-47 locale per task. If the user picks `en-US` and speaks Russian, it hesitates and produces garbage. There is no Apple-native auto-detect, and no native code-switching support.
2. **No reference audio.** We test ASR with the user speaking into the phone. That's fine for smoke testing but useless for regressions, accuracy benchmarks, or marketing claims.

This doc specifies how we fix both.

---

## 2. Vision

### 2.1 Multilingual recognition with hardware-aware tiers

Devices vary. We advertise our recognition strategy as a **tier** in `/capabilities`, and clients (and tests) can rely on it:

| Tier | Strategy | Required HW | Behavior |
|---|---|---|---|
| `single` | One `SFSpeechRecognizer` for one user-picked locale | Any iOS 17+ device | Today's behavior |
| `parallel` | N `SFSpeechRecognizer` instances run on the same audio buffer; pick highest-confidence transcript per utterance | Any iOS 17+ device with ≥3GB RAM | Handles utterance-level switches |
| `whisper` | On-device Whisper via Core ML; auto language ID + true mid-sentence code-switching | A14+ chip, ≥6GB RAM, Neural Engine | The "intelligent" tier |

Each node *also* advertises which tier it actually selected, plus its candidate languages and the Whisper model size if applicable.

`/capabilities` `_detail.speech_to_text` shape (extension of today's):

```json
{
  "available": true,
  "on_device": true,
  "tier": "whisper",                          // "single" | "parallel" | "whisper"
  "supported_locales": ["ar-SA", ...],        // SFSpeechRecognizer locales (always)
  "candidate_languages": ["en-US","ru-RU","fr-FR"],  // user's selected set (parallel/whisper)
  "whisper": {                                // present iff tier == "whisper"
    "model": "small.en-multi",
    "model_bytes": 244000000,
    "auto_language_id": true,
    "code_switching": true
  }
}
```

### 2.2 Canonic audio: Azure-generated tongue-twister fixtures

We commit a small set of **reference WAVs**, one per locale, generated by **Azure Cognitive Services Neural TTS** (the same engine pronunco uses today). Each WAV is a famously hard tongue-twister in that language. These give us:

- **Regression tests** that don't require a human to speak into a phone.
- **Accuracy benchmarks** comparable across nodes (iOS, Android, server-side Whisper).
- **A real marketing line:** *"iHomeNerd is tested on the world's hardest phrases in seven languages — all on-device."*

Twister set (initial):

| Locale | Twister |
|---|---|
| `en-US` | Peter Piper picked a peck of pickled peppers |
| `ru-RU` | Карл у Клары украл кораллы, а Клара у Карла украла кларнет |
| `fr-FR` | Les chaussettes de l'archiduchesse sont-elles sèches, archi-sèches? |
| `de-DE` | Fischers Fritz fischt frische Fische, frische Fische fischt Fischers Fritz |
| `it-IT` | Trentatré trentini entrarono a Trento, tutti e trentatré trotterellando |
| `es-ES` | Tres tristes tigres tragaban trigo en un trigal |
| `zh-CN` | 四是四，十是十，十四是十四，四十是四十 |

Easy to extend. PRs welcome.

## 3. Spec — Azure fixture generator

### 3.1 What it produces

```
mobile/testing/fixtures/tongue_twisters/
  en-US.wav
  ru-RU.wav
  fr-FR.wav
  de-DE.wav
  it-IT.wav
  es-ES.wav
  zh-CN.wav
  manifest.json   # twister text, voice name, region, sample rate, sha256
```

### 3.2 What it consumes

- Azure Speech key + region from a path-resolvable secret. Recommended order of lookup:
  1. `AZURE_SPEECH_KEY` / `AZURE_SPEECH_REGION` env vars
  2. pronunco's existing config file (whatever path it uses on the Dell — owner of the Dell knows)
  3. Fail with a clear error pointing at this doc

### 3.3 Where it lives

`mobile/testing/scripts/generate_tongue_twisters.py` (or `.ts` — implementer's choice). Self-contained. Uses Azure REST API directly, no SDK, so it has zero install footprint beyond the language runtime.

### 3.4 Voices

Use Azure Neural voices. Per locale, default to a recognizable neutral voice (e.g. `en-US-JennyNeural`, `ru-RU-SvetlanaNeural`, etc.). Voice names go into `manifest.json` so we can audit later.

### 3.5 Output format

- 16-bit PCM, 16 kHz, mono. (Apple Speech and Whisper both accept this directly.)
- `.wav` container, no compression.
- Filename = locale BCP-47.

### 3.6 Determinism

Generation is **not** deterministic across Azure model upgrades. The script must:
- Pin the voice name explicitly (no "default voice").
- Record voice + region + script run timestamp in `manifest.json`.
- Always overwrite WAVs on rerun (no silent "already exists" skip). If you don't want regenerate, don't run the script.

### 3.7 Commit policy

- WAVs **are** committed to the repo. They're small (~50–200 KB each, ~1 MB total).
- `manifest.json` is committed.
- Azure keys are **never** committed.

## 4. Spec — iOS multilingual ASR

### 4.1 Tier detection

At `NodeRuntime.start()`, inside `CapabilityHost.snapshot()`:

```swift
enum SpeechTier { case single, parallel, whisper }

static func detectTier() -> SpeechTier {
    let mem = ProcessInfo.processInfo.physicalMemory
    let hasWhisperHW = mem >= 6 * 1024 * 1024 * 1024  // ≥6GB
        && hasNeuralEngine()                            // A14+ proxy
        && WhisperModel.bundled.exists()
    if hasWhisperHW { return .whisper }
    if mem >= 3 * 1024 * 1024 * 1024 { return .parallel }
    return .single
}
```

`hasNeuralEngine()` can be probed via `MLComputeUnits.cpuAndNeuralEngine` availability + a quick warm-up of a no-op MLModel, or device-model lookup against a known list (A14 / M-series). Implementer's choice; document the heuristic in the code.

### 4.2 Parallel mode (Tier `parallel`)

- Listen tab shows a **multi-select** language picker (replaces today's single picker) with the user's "candidate languages." Default to top 3 from `Locale.preferredLanguages` plus `en-US`.
- Up to **4 recognizers** run in parallel. We `append(buffer:)` the same audio buffer to all `SFSpeechAudioBufferRecognitionRequest` instances.
- For each utterance, when results come in, score by `bestTranscription.segments.map(\.confidence).reduce(0,+)`. Display the highest-scoring transcript and quietly stop the others on `isFinal`.
- UI: show a small subtle badge `[ru-RU]` or similar next to the transcript indicating which recognizer won. Helpful and on-brand.

### 4.3 Whisper mode (Tier `whisper`)

- Bundle a Core ML Whisper model (start with `small`, multilingual, ~244 MB). App size grows; this is fine — gated behind tier eligibility, and we can offer it as a separate "Multilingual Pack" download later.
- Run via WhisperKit or a hand-rolled pipeline (decision deferred — start with WhisperKit if license allows).
- Whisper's output includes per-segment language. Display segments inline with language tags when they switch mid-utterance.
- Whisper does not stream as cleanly as `SFSpeechRecognizer`. For now, run it on **finalized audio chunks** (every ~3 seconds of voice activity) rather than continuously. Live partials come from a parallel `SFSpeechRecognizer` running underneath in the user's primary locale; Whisper "corrects" on segment boundaries.

### 4.4 UI on the Listen tab

- **Mode toggle** at top: `Single | Parallel | Whisper` — disabled options grey-out with a tooltip explaining the HW requirement.
- **Language set picker** (parallel mode) — chip-style multi-select.
- **Live transcript** with optional per-segment locale badge.
- **Status line** still shows tier + on-device/server in mono font, as today.

## 6. Open questions

1. **Whisper distribution.** Bundling 244 MB in the app is doable but gross. Alternatives: ship empty-shell app, download model on first Whisper-tier launch, cache in App Support. Decide before we ship Whisper publicly.
2. **iPad / Mac.** Same tiering logic should apply. M-series Macs are unambiguously `whisper` tier.
3. **Per-segment language tags in transcript UI.** Inline badges look nice but may clutter. A/B with the user before committing to a final design.
4. **Privacy copy.** Whisper is fully on-device — confirm the Listen-tab footer language reflects that distinction vs. `SFSpeechRecognizer`'s server-fallback behavior.
